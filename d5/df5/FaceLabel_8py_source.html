15 """
16
17 SetLabels is an action set which is automatically merged into all ExaHyPE2
18 steps by the project. There's nothing for the user solvers to be done here.
19
20 There are four fields which have a value per side of the face:
21
22 - Updated: Boolean[2]
23 Shows if the corresponding adjacent cell (left vs right) has performed a
24 time step.
25 - UpdatedTimeStamp: Double[2]
26 Used only by the Finite Volumes solver so far to store a new time stamp
27 temporarily which is later on rolled over into the real time stamp.
28 - NewTimeStamp: Double[2]
29 New (and also current time stamp). This time stamp determines for example
30 if you want to do a local time step or not.
31 - OldTimeStamp: Double[2]
32 Time stamp of previous time step. The old time stamp is required for the
33 in-time interpolation, e.g.
34
35 In addition there are three fields that are in common to both faces:
36
37 - Boundary: Boolean
38 Shows whether the cell is at a domain boundary. This is set upon creation
39 by checking one's position in relation to the domain boundaries.
40 - IsHanging: Boolean
41 This states whether the face is a hanging face, i.e. whether the face
42 is adjacent to a cell with a coarser refinement level and therefore there
43 is no face directly "across" from this one for it to merge with.
44 - AboveHanging: Boolean
45 This states whether the face is above a hanging face, i.e. whether the
46 face is adjacent to a cell with a finer refinement level and therefore
47 the face directly "across" from this one will likely contain no useful
48 information, meaning said information must be obtained from the finer
49 level underneath this one.
50
51 ## Initialisation
52
53 The initialisation sets the boundary flag if the face is on the boundary.
54 It also sets the time stamps all to zero. This is actually a tricky thing
55 to do, so I'm not sure if it is a good idea: the DG solvers for example
56 require the face time stamps to be set properly. The Finite Volumes solver
57 in return benefits if it is not (properly) initialised, as it then is
58 simpler to spot inconsistencies and missing data updates.
59
60 ## IsHanging and AboveHanging
61
62 IsHanging is a redundant information. If a face is hanging, it is not held
63 persistently. Consequently, we get a createHangingFace() in each and every
64 mesh traversal and hence know implicitly that the face is hanging. The flag
65 therefore replicates information which is implicitly available anyway.
66
67 AboveHanging is stored persistently. At the moment, it is only used by
68 ADER-DG solvers. Most solvers should be able to work without this additional
69 information. For ADER-DG, it is used to ensure that Riemann-solves
70 are performed on fine faces and not on coarse faces, as the later would
71 cause unphysical transfer of information which lead to uncontrolled
72 oscillations.
73
74 AboveHanging is historical information. If the mesh is just about to refine,
75 then the flag might not be set even though we literally just add new
76 "subfaces". If a mesh coarsens, the flag might be set, even though we
77 literally just remove the adjacen fine grid cells. If you need to know what
78 happened in this very iteration, you can read out the flag NewAboveHanging,
79 but this one is only valid in touchFaceLastTime(), i.e. while we ascend in
80 the tree.
81
82 Please consult peano4::datamanagement::FaceMarger::hasBeenRefined() in this
83 context, too.
84
85 """
