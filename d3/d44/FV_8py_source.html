16class FV(object):
17 """!
18 Abstract finite volume solver step sizes that works on patch-based AMR with a halo layer of one.
19
20 FV is the base class of all FV solvers. It defines what kind of action sets do
21 exist, i.e., what in principle can be done while we run through the grid. It also
22 provides some kind of very basic infrastructure, i.e., what the name of a solver
23 is, what data is to be held per face or cell, or where in the multiscale mesh
24 we actually have to hold data. The FV class cannot/should not be instantiated.
25
26 A finite volume solver in ExaHyPE 2 (actually any solver) is first of all a
27 static collection of data logically tied to grid entities and some operations
28 over the mesh that are just there. All data and action sets however have guards,
29 i.e., boolean guards that define
30
31 - are data to be stored in-between traversals,
32 - are action sets really invoked in a particular traversal or can they be
33 skipped.
34
35 In the baseline FV class, these guards are usually set to default values.
36 That is, all action sets per time step are invoked always (the guard is
37 true) and all data are always stored on the finest mesh. If you want to alter
38 this behaviour, i.e., store data not always or skip steps, you can overwrite
39 the corresponding attributes of the attributes, i.e., the guards of the
40 associated data.
41
42 See the discussion on "Control flow between this class and subclasses" below.
43
44 ## Parallelisation
45
46 I do equip both Q and NewQ with proper merge routines. However, all merge guards
47 set to "never" by default. If you need some data exchange, you have to activate
48 them manually.
49
50 ## Control flow between this class and subclasses
51
52 There are three key routines: the constructor, create_data_structures() and
53 create_action_sets(). The constructor sets some global variables (such as the
54 name) and then invokes the other two routines.
55
56 create_data_structures() establishes all the data structures tied to the
57 grid entities. It also sets some properties of these data such as the patch
58 size, e.g. If you want to add additional data (such as additional quantities
59 per cell) or if you want to alter the configuration of data tied to grid
60 entities, you should redefine create_data_structures(). However, any
61 subclass still should call FV's create_data_structures() - or the create_data_structures()
62 of the SingleSweep. This will ensure that the
63 baseline configuration of all data is in place. After that, you can modify
64 the properties.
65
66 create_action_sets() establishes the action sets, i.e. activities that are to
67 be triggered whenever you run a time step, you plot, you initialise the grid.
68
69 Both create_data_structures() and create_action_sets() add attributes to the
70 FV class. See self._patch for example within create_action_sets(). These
71 attributes have guards such as self._action_set_initial_conditions.guard.
72 These guards are set to defaults in FV. It is basically the job of SingleSweep
73 or any other subclass - they determine when which data are used - to reset these
74 guards from a default to something tailored to the particular data flow.
75
76 If you want to redefine when data is stored or operations are invoked, overwrite
77 create_data_structures(), and call the superclass. This is what happens:
78
79 - SingleSweep passes on the call to FV.create_data_structures().
80 This call ensures that all the data structures are in place. Then it returns.
81 - SingleSweep's create_data_structures() then sets the
82 guard, i.e. they configure when data is stored or used.
83 - Finally, your own overwrite of create_data_structures() can augment the
84 data structures (which are now in place and called at the right time) with
85 information what it actually shall do.

564 @abstractmethod
565 def create_data_structures(self):
566 """!
567
568 Create data structures required by all Finite Volume solvers
569
570 Recall in subclasses if you wanna change the number of unknowns
571 or auxiliary variables. See class description's subsection on
572 data flow.
573
574 :: Call order and ownership
575
576 This operation can be called multiple times. However, only the very
577 last call matters. All previous calls are wiped out.
578
579 If you have a hierarchy of solvers, every create_data_structures()
580 should first(!) call its parent version. This way, you always ensure
581 that all data are in place before you continue to alter the more
582 specialised versions. So it is (logically) a top-down (general to
583 specialised) run through all create_data_structures() variants
584 within the inheritance tree.
585
586 :: Arguments
587
588 _patch: Patch (NxNxN)
589 Actual patch data. We use Finite Volumes, so this is
590 always the current snapshot, i.e. the valid data at one point.
591
592 _patch_overlap_old, _patch_overlap_new: Patch (2xNxN)
593 This is a copy/excerpt from the two adjacent finite volume
594 snapshots plus the old data as backup. If I want to implement
595 local timestepping, I don't have to backup the whole patch
596 (see _patch), but I need a backup of the face data to be able
597 to interpolate in time.
598
599 _patch_overlap_update: Patch (2xNxN)
600 This is the new update. After the time step, I roll this
601 information over into _patch_overlap_new, while I backup the
602 previous _patch_overlap_new into _patch_overlap_old. If I
603 worked with regular meshes only, I would not need this update
604 field and could work directly with _patch_overlap_new. However,
605 AMR requires me to accumulate data within new while I need
606 the new and old data temporarily. Therefore, I employ this
607 accumulation/roll-over data which usually is not stored
608 persistently.
609 """

785 """!
786 Create all the action sets
787
788 Overwrite in subclasses if you wanna create different
789 action sets. Subclasses also might redefine the invocation order.
790
791 ## Call order and ownership
792
793 This operation can be called multiple times. However, only the very
794 last call matters. All previous calls are wiped out.
795
796 If you have a hierarchy of solvers, every create_data_structures()
797 should first(!) call its parent version. This way, you always ensure
798 that all data are in place before you continue to alter the more
799 specialised versions. So it is (logically) a top-down (general to
800 specialised) run through all create_data_structures() variants
801 within the inheritance tree.
802
803 ## Recreation vs backup (state discussion)
804
805 We faced some issues with action sets that should not be
806 overwritten. For example, the postprocessing should not be overwritten
807 as users might want to set it and then later on reset the number of
808 unknowns, e.g. In this case, you would loose your postprocessing if
809 create_action_sets() recreated them. So I decided to make an exception
810 here: the postprocessing step is never overwritten by the standard
811 classes.
812
813 There are further action sets which have a state, which users might
814 want to alter. The most prominent one is the AMR routines, where users
815 often alter the interpolation and restriction scheme. Here, things are

1089 def add_actions_to_init_grid(self, step, restart_from_checkpoint=False):
1090 """!
1091 Add all the action sets to init grid
1092
1093 The AMR stuff has to be the very first thing. Actually, the AMR routines'
1094 interpolation doesn't play any role here. But the restriction indeed is
1095 very important, as we have to get the face data for BCs et al. The action
1096 set order is inverted while we ascend within the tree again. Therefore, we
1097 add the AMR action set first which means it will be called last when we go
1098 from fine to coarse levels within the tree.
1099
1100 ## Ordering
1101
1102 The order of the action sets is preserved throughout the steps down within
1103 the tree hierarchy. It is inverted throughout the backrolling.
1104
1105 This is what we want to achieve:
1106
1107 - Project solution onto faces. This happens in touchCellLastTime(). See
1108 exahype2.solvers.fv.actionsets.ProjectPatchOntoFaces for comments.
1109 The project will end up in QUpdate.
1110 - Roll updates over on the faces from QUpdate into Q_new. This is done
1111 by RollOverUpdateFace, which requires in turn that the getUpdated()
1112 flag is set. As the roll-over plugs into touchFaceLastTime(), it will
1113 always be called after the projection, since the latter is a cell
1114 operation.
1115 - Copy new face data Q_new into old face data Q_old, as this is the
1116 initial sweep, i.e. the old face data otherwise might hold rubbish.
1117 This is a backup operation realised through the action set
1118 BackupPatchOverlap. This one plugs into touchFaceLastTime() too.
1119 Therefore, it is important that its priority is smaller than the one
1120 of the roll-over, since we invert the call order for the touch-last
1121 events.
1122 - Restrict the data to the coarser level if we are on a hanging face.
1123 - The postprocessing is the very last thing. It does not matter if it
1124 plugs into touchCellFirstTime() or touchCellLastTime(). As we invert the
1125 order when we backtrace, the postprocessing will have completed before
1126 we invoke _action_set_project_patch_onto_faces, i.e. any update will
1127 be projected onto the faces.
1128 """
1129 assert (
1130 self._action_set_copy_new_faces_onto_old_faces.descend_invocation_order
1131 < self._action_set_roll_over_update_of_faces.descend_invocation_order
1132 )

1135 self._action_set_initial_conditions = (
1136 exahype2.solvers.fv.actionsets.InitialCondition(
1137 self, self._store_cell_data_default_guard(), "true", restart_from_checkpoint=True
1138 )
1139 )

1141 step.add_action_set(self._action_set_preprocess_solution)
1142 step.add_action_set(
1143 self._action_set_couple_resolution_transitions_and_handle_dynamic_mesh_refinement
1144 )
1145 step.add_action_set(self._action_set_copy_new_faces_onto_old_faces)
1146 step.add_action_set(self._action_set_roll_over_update_of_faces)
1147 step.add_action_set(self._action_set_initial_conditions)
1148 step.add_action_set(self._action_set_project_patch_onto_faces)
1149 step.add_action_set(self._action_set_update_face_label)

1665 @preprocess_reconstructed_patch.setter
1666 def preprocess_reconstructed_patch(self, kernel):
1667 """!
1668 Set a new processing kernel
1669
1670 Before the actual update, the numerical solver takes the actual solution
1671 including all auxiliary variables and copies it into another array which
1672 is calls reconstructedPatch (in FD4, this field is called oldQWithHalo).
1673 This array is bigger than the actual patch data, as it also holds the
1674 halo. Once the solution is copied over, the kernel writes in the halo
1675 data using data from the faces. This reconstructed object is now passed
1676 into the compute kernel, which derives new values for all evolution
1677 variables. That is, the compute kernels' output does not hold the auxiliary
1678 variables anymore, as they are not subject to the PDE. The new variables
1679 are eventually written back into the real data, where they replace the
1680 old quantities (the auxiliary variables in there remain untouched).
1681 After the compute kernel has terminated, the reconstructed data including
1682 the halo is thrown away.
1683
1684 ## Auxiliary variables
1685
1686 If you alter the auxiliary variables (material parameters) in the patch
1687 preparation, this is absolutely fine, but the change will not be committed
1688 into the real data. It is only there temporarily while the actual patch
1689 will continue to hold the original material parameters.
1690
1691 You can alter the actual patch data Q in the preprocessing. If this
1692 actual data are evolution (PDE) parameters, these changes however will
1693 be overwritten by the compute kernel. If you alter auxiliary variables
1694 within the preprocessing within the patch (Q), then this is fine, but the
1695 changes will not be visible to the reconstructed patch immediately. They
1696 will be visible there in the next time step. So you might be well-advised
1697 to alter the auxiliary variables in both the patch data and the
1698 reconstructed data.
1699
1700 ## Access
1701
1702 We recommend that you use the AoS enumerator exahype2::enumerator::AoSLexicographicEnumerator.
1703 This one has to be given 1 as number of cells/patches, it requires the size
1704 of the patch, its halo size, you have to pass it the unknowns and the correct
1705 number of auxiliary parameters. If you want the preprocessing also to alter
1706 the actual input, you need a second enumerator. In this second one, all
1707 parameters are the same besides the halo size, which you have to set to 0:
1708 The input patch has no halo. This halo is only added temporarily in the
1709 reconstruction step.
1710 """
1711 self._preprocess_reconstructed_patch = kernel
1712 self.create_data_structures()
1713 self.create_action_sets()

1736 enforceCCZ4constraints( newQ+index );
1737 index += {{NUMBER_OF_UNKNOWNS}} + {{NUMBER_OF_AUXILIARY_VARIABLES}};

1743

1753 within your injected code. Furthermore, you have the following two variables
1754 available:
1755

1757 array). The patch is not supplemented by a halo layer.
1758 - marker An instance of the CellMarker which identifies the cell.
1759
1760 In ExaHyPE's finite volume solvers, there are two different ways how to inject
1761 postprocessing: You can either define your own postprocessing step by setting
1762 _action_set_postprocess_solution, or you can use this setter to inject some
1763 source code. The two options are not equivalent: The snippet injected here is

1766 update a cell.
