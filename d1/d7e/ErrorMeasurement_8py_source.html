8class ErrorMeasurement:
9 """!
10 Error measurement tool for ExaHyPE 2 solvers, this is the base class that must
11 be specialised for different solvers.
12
13 Used by calling the corresponding specialisation by calling it using the solver
14 instance as an argument. For an example, see /applications/exahype/aderdg
15 i.e. ErrorMeasurement(solver=theSolver)
16
17 When called, it adds an additional function analyticalSolution() to the user-defined
18 code snippets, through which the user can define what the solution should be.
19
20 The errormeasurement also adds a postprocessing step to the corresponding solver,
21 which computes the L0, L2 and L_inf error between the computed solution and the
22 analytical solution at each data point over the domain. These are then summed up
23 over the entire volume to get the integrated errors over the entire domain.
24
25 Finally, these are added to a GlobalDatabase object by the global master thread, and
26 outputted to csv files at regular intervals.
27
28 @parameters:
29
30 solver: the solver for which the errors should be measured
31
32 error_measurement_implementation: allows one to specify the analytical
33 solution directly instead of manually filling it in the user-defined
34 code snippets, analogously to fluxes and such in the solvers.
35
36 delta_between_database_flushes: how often should the entire database be
37 printed to a file. By default the database is only printed once at the
38 end of the entire simulation, but this parameters can be used to produce
39 intermediary results
40
41 output_file_name: specify what the output .csv files should be named
42
43 data_delta_between_snapshots: which data deltas should be used for the
44 database, i.e. if a measurement already exists for the given timestamp
45 but the difference is greater than this value, the existing measurement
46 is replaced.
47
48 time_delta_between_snapshots: how often a new measurement should be added
49 to the database. By default the database is updated every timestep, but
50 using this parameter it can be set to only happen at regular intervals,
51 analogously to plotting in ExaHyPE2.
52
53 clear_database_after_flush: If the parameter "delta_between_database_flushes"
54 has been set and several database flushes are going to occur, this
55 parameter can be used to flush the database between each flush in order
56 to avoid printing redundant information.
57
58 """
