Detailed Description

Most efficient solver we have for this type of problem class.

You can find each of the action sets here: solvers.api.actionsets.DGSolver.InitHigherOrderDofs.InitHigherOrderDofs

The text below specifies the semantics and behaviour of this matrix-free DG solver. A tutorial running step by step through the steps how to use it can be found in the tutorials section. The solver in general tackles a PDE

\( \mathcal{L}(u) = f \)

resulting in a linear equation system

\( Au = b. \)

Throughout the discussion, we unfold the compute steps over the compute mesh.

Storage

This solver holds \( K^c (p+1)^d \) unknowns (doubles) per cell for the solution, \( K^c (p+1)^d \) per cell for the right-hand side, and the same number for the residual. Holding the residual in the cell allows us to separate some algorithm steps into different action sets. In particular, we project the face solution back into the cells after updating the cell solution (in separate sweeps). These are the primary variables, i.e. those directly resulting from the PDE formulation. The correspond to the unknowns directly from \( u \) and the right-hand side.

Further to that, we hold auxiliary variables on the faces: Our Riemann solver accepts the left and the right solution. Here, each degree of freedom has \( K^f_{f \gets c} \) unknowns. It will return a result (the flux), which has \( K^f \) unknowns. Per face node, we hence hold, in total, \( ( K^f + 2K^f_{f \gets c} ) (p+1)^{d-1} \) unknowns. That is the solution for each unknown per face node, as well as a projection from the left and right for each unknown per cell. \( K^c \) and \( K^f \) can be different, but often they are the same. The user can choose how many of the \( K^c \) to project onto the face by modifying \( K^f_{f \gets c} \).

Operators

With the auxiliary variables, in place, we can write down the DG problem as

\( \left( \begin{array}{cccc} A^{cc} & & & P_{c \gets f} \\ P_{f \gets c} & id & & \\ P_{f \gets c} & & id \\ & id & id & A^{ff} \end{array} \right) \left( \begin{array}{c} u^c \\ u^+ \\ u^- \\ q \end{array} \right) = \left( \begin{array}{c} M b \\ 0 \\ 0 \\ 0 \end{array} \right) \)

in line with xxxx. The \( q \) is the result of the Riemann solver.

Solver

We now realise a Block-Jacobi smoother subject to two relaxation parameters \( \omega _c, \omega _f \in [0,1] \). We tackle the first line in the block system with one smoothing step of a stationary iterative scheme, solve two and three exact (as they are trivial with the identity on the diagonal) and apply Jacobi to the last row again. As the first and last block are updated in parallel by the Jacobi scheme, the resuling algorithm is globally a Jacobi relaxation, too.

  • Per cell, we tackle the equation \( A^{cc} u^c + P_{c \gets f} q = M b \) which yields the update rule

    \begin{eqnarray*} u|_c & \gets & u|_c + \omega _c \tilde {A}^{-1}\left( Mb - A^{cc} u^c - P_{c \gets f} q \right). \end{eqnarray*}

    Here, the \( q \) is the solution of the PDE term on the faces which we assume to be fixed. At this point, we are quite generic and dump the responsibility onto the user to give us an approximation of \( A_{cc} \) that we can invert. Notably, we highlight that the user may give us \( \tilde{A} = A^{cc} \) in a DG sense, as we cannot invert \( A^{cc} \).
  • Per face, we use a plain Jacobi. We assume that we have an explicit expression of the Riemann solution (simple averaging would be an example) and hence \( \hat A = id \). We consequently get

    \begin{eqnarray*} q|_f & \gets & q|_f + \omega _f \left( \mathcal{R}(u^-, u^+) - q|_f \right) \\ & = & (1-\omega _f) q|_f + \omega _f \mathcal{R}(u^-, u^+). \end{eqnarray*}

Algorithmical blueprint

  1. Enter cell: Whenever a cell is entered, we create a temporary \( K^c(p+1)^d \) array (on the heap).
    1. Into this temporary array \( r \), we write \( Mb \). This is the mass matrix (right-hand side matrix of the class constructor, as we are not restricted to a traditional mass matrix) times the right-hand sides within the cell. So \( r \gets Mb. \)
    2. Once complete, we substract \( P^{c \gets f} q|_f \). That is, we take the solution on the face and we project it (cell from face) onto the cell, where we substract it from the previous \( r \) value. We also do the same with the cell-to-cell couplings. So: \( r \gets Mb - Pq|_f -A^{cc}u^c. \) We also do the same with the cell couplings
    3. The third step is where the MFlops come from: We take the user's approximation of the cell matrix and explicitly invert it. For a linear, invariant matrix \( A \), we could do so once, but the solver itself makes no assumption if the matrix is invariant or not. So it inverts it every step and uses BLAS routines or this. The result is multiplied with the residual as accumulated so far: \( r \gets \tilde{A}^{-1} \left( Mb - Pq|_f -A^{cc}u^c \right). \)
    4. We run over the solution within the cell and scale each entry with \( (1-\omega _c) \). If \( \omega _c =1 \), this means we set the entries to zero. In general, the entries however will only be damped. After that, we add the \( r \) values to each point. This gives us a new solution guess.
    5. We project the updated solution \( u|_c \) onto the faces, i.e. obtain a new \( u|_f^{\pm } \). This is realised within the action set ProjectOntoFaces.

  2. touchFaceFirstTime(): Whenever a face is touched, we solve the Riemann problem and update the Riemann guess accordingly. Initially, we assume that the \( q \) entries on the face are all set to zero.

    1. We evaluate per node on the face the expression \( \mathcal{R}(u^-,u^+) \). For this, we loop over the \( (p+1)^{d-1} \) points and call the C++ solver's routine. The user can thus implement the actual Riemann solver themself.
    2. We take the result and update the value on the face according to \( q|_f \gets (1-\omega _f) q|_f + \omega _f \mathcal{R}(u^-, u^+). \)

    All of this functionality can be found in the subpackage actionsets.UpdateResidual.

How does it work?

Python stuff

There is a very simple Poisson solver tutorial in the tutorials section which illustrates how to use this solver.

The user needs to supply a few crucial matrices, referred to in the maths above. These are noted and their shapes discussed in the parameters section of the documentation of this class.

Under the hood

Inside the cells we store \( K^c (p+1)^{d} \) values for the solution, and the same number for the rhs and residual. We store the residual in the cell so that we can calculate the update to the cell solution separately from the projection of face solution back into the cells, but crucially are able to update the global residual before we apply the preconditioner. On each face we store \( K^f (p+1)^{d-1} \) values for the solution, and then \( 2K^f_{f \gets c} (p+1)^{d-1} \) for the projections in a separate array. We note that this factor of two comes from the fact that we have projections from cells on both the left and right. The user is free to choose how many variables from the cell they want to project onto the face. face_from_cell\( \mathbb{R}^{2d * 2K^f_{f \gets c} * (p+1)^{d-1} \times K^c * (p+1)^{d}}\) This matrix projects the cell solution onto all of the faces. Note the extra factor of \( 2d \). The structure of this matrix is quite important. The rows should be structured in blocks of \( 2K^f_{f \gets c} * (p+1)^{d-1} \), ie one for each face, in the familiar Peano ordering. We note that this is the same as the number of projections we store per face. We note that, since the 0th face has left-side projections which should be set outside of the cell, that at least half of the rows of the matrix should be empty. We also note that each face contains multiple nodes, and they should be ordered sequentially in the typical Peano fashion, i.e. greedily along each coordinate axis until full, starting from the negative. For each face, with two nodes, with two values (q and u) being projected onto the face, the rows should be ordered as follows:
  1. Negative projection of q onto node 0
  2. Negative projection of q onto node 1
  3. Negative projection of u onto node 0
  4. Negative projection of u onto node 1
  5. Positive projection of q onto node 0
  6. Positive projection of q onto node 1
  7. Positive projection of u onto node 0
  8. Positive projection of u onto node 1 In two dimensions, all of the negative projections should be 0 on faces 0 and 1. Ditto for the positive projections on faces 2 and 3.
cell_from_face\( \mathbb{R}^{K^c * (p+1)^{d} \times 2d K^f * (p+1)^{d-1}}\) This matrix projects the face solution onto the cells. Note the extra factor of \( 2d \), similar to face_from_cell matrix. Some clever indexing is required in the backend here. The enumeration of the cols should follow a similar pattern to the rows of face_from_cell.
