32class VolumeWisePreprocessSolution(AbstractFVActionSet):
33 """!
34 Run over solution volume by volume
35
36 PreprocessSolution differs from other action sets, as I only create it once. See
37 FV.create_action_sets(). Once the instance does exist, you can tailor it. But you
38 can also completely overwrite it.
39
40 See also FV.postprocess_updated_patch(self, kernel) for a discussion what options
41 ExaHyPE offers for postprocessing. This volume-wise postprocessing might be too
42 heavy-weight for some applications.
43
44 The postprocessing plugs into touch cell last time. It is the last action set added
45 by the default implementation. Therefore, it is the first action set of which
46 touchCellLastTime() is called. The reason why I plug into the leaving of the cell
47 is that some solvers may add further action sets to the solve. Enclave tasking for
48 example adds the merger as additional action set. These action sets plug into
49 touchCellFirstTime() - and consequently their first time is called after the
50 postprocessing's first time. By making the postprocessing use touchCellLastTime(),
51 I ensure that any additional action set added by a subclass can still precede the
52 postprocessing.
53 """
54 def __init__(self,solver):
55 AbstractFVActionSet.__init__(self,solver)
56 self.guard="true"
57 self._compute_kernel=""
58 self.descend_invocation_orderdescend_invocation_order=1

68

70 C/C++ code snippet
71 """
72 self._compute_kernel += """

80 {{PREDICATE}}

82 logTraceIn("touchCellFirstTime(...)");

88 auto h = ::exahype2::fv::getVolumeSize(marker.h(), {{NUMBER_OF_VOLUMES_PER_AXIS}});

97 def get_body_of_operation(self,operation_name):
98 result = ""
99 if operation_name==peano4.solversteps.ActionSet.OPERATION_TOUCH_CELL_LAST_TIME:
100 d = {}
