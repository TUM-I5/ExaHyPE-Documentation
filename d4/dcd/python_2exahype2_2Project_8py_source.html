17
18 Represents an ExaHyPE 2 project. An ExaHyPE 2 project is a Peano 4

358 def set_restart_from_checkpoint(self, expected_restart_timestamp=0, checkpoint_path="./"):
359 '''
360 set checkpointing. The function accept the corresponding peano
361 patch file and create corresponding arrays recording the checkpoint
362 file names.
363
364 @param restart_timestamp float
365 targeting restart timestamp, the function will find the first
366 checkpointed timestamp right after this input time. Thus the final
367 restarting timestamp may not be equal to this specified time.
368
369 @param checkpoint_file string
370 the location of the checkpoint meta-file
371 '''
372
373 self._restart_from_checkpoint = True
374 self._restart_timestamp = expected_restart_timestamp
375 self._checkpoint_path = checkpoint_path
376
377 for solver in self._solvers:
378 checkpoint_meta_file=checkpoint_path+"checkpoint-"+solver._name+".peano-patch-file"
379 recording_current_snapshot=False
380 solver_file_collection=[]
381 with open(checkpoint_meta_file,'r') as file:
382 for line in file:
383 if line.strip().split()==[]: continue
384 if line.strip().split()[0] == "timestamp" and float(line.strip().split()[1])>=self._restart_timestamp:
385 self._restart_timestamp = float(line.strip().split()[1])
386 recording_current_snapshot=True
387 if line.strip().split()[0] == "include" and recording_current_snapshot==True:
388 solver_file_collection.append( line.strip().split()[1] )
389 if line.strip().split()[0] == "end" and recording_current_snapshot==True:
390 break
391 if recording_current_snapshot:
392 self._solvers_file_collection.append(solver_file_collection)

452 def add_tracer(
453 self,
454 name,
455 particle_attributes,
456 plot=True,
457 sort_descend_invocation_order_within_action_sets=-1,
458 plot_descend_invocation_order_within_action_sets=65536,
459 ):
460 """!
461
462 Add a tracer to the project
463
464 Tracers have to be the very last thing you add to your project.
465 At this point, all solvers have to be added to the project.
466 However, you have to add the tracers prior to any user-defined
467 routines. See docu of init_new_user_defined_algorithmic_step().
468
469 We rely on the particle toolbox. Therefore, our particles have
470 already a position x, a search radius (which we don't usually
471 use here), and a (parallel) state. The search radius determines
472 how Peano sorts particles within the tree, and we can set it to
473 zero here. This ensures that the particle has no real volume and
474 thus is always sieved through to the finest grid level.
475
476 @see peano4.toolbox.particles.Particle
477
478
479 ## Projection of data onto particles and database dumping
480
481 The projection onto the particle has to be realised separately, as
482 it depends on the solver variant chosen. The same holds for the
483 dump of particle data into a database if this shall happen
484 on-the-fly.
485
486 Most bigger applications add their solvers an add_tracer() operation,
487 which takes all of this domain knowledge into account. This
488 add_tracer() then forwards to this routine, adding further
489 behaviour.
490
491 See the class application.exahype2.ccz.CCZ4Solver and its derived
492 variants for examples.
493
494
495 ## Moving particles
496
497 Tracers in ExaHyPE can move, though they do not have to move. Even if
498 they move, they never have a native velocity, i.e. their velocity will
499 always be a projection of an ExaHyPE solution. We are talking about
500 tracers after all and not about a proper PIC approach. Due to this
501 constraint, we do not need a velocity vector. However, we need an
502 additional enumeration which clarifies if a
503 particle has moved already or not. This enumeration is needed by the
504 particle update routines. Obviously, nothing stops you from adding
505 a separate velocity field to the tracer object and thus to redefine
506 this behaviour.
507
508
509 ## ExaHyPE-specific particle attributes
510
511 We add each
512 particle a number which is a tuple consisting of the tree number
513 (on which tree has a particle been generated plus what was the
514 tree-local number of this particle at the time). Through this
515 tuple, we can track tracers later on even if they move across
516 domains.
517
518 We can plot the data - if there is data to be tracked - directly,
519 as it is a large double array. For the index, we have to type cast
520 the data, as this is a tuple of integers, and Peano's VTK plotter
521 supports only double vectors.
522
523 @see peano4.toolbox.particles.PlotParticlesInVTKFormat.add_attribute_to_plot()
524
525
526 ## Action set order
527
528 We add the particle ordering with descend order -1. This should do the job
529 for most setups, but we might be wrong. The following action sets are
530 automatically added:
531
532 - UpdateParallelState is added to each and every time step and plotting
533 sweep. As we also add
534 - peano4.toolbox.particles.api.UpdateParticleGridAssociation_LiftDrop to each and every time step,
535 particles can move at any point. We always resort them immediately.
536
537 Both action sets are assigned the standard priority of sort_descend_invocation_order_within_action_sets.
538
539
540 ## Arguments
541
542 name: String
543 Has to be a unique name for this tracer
544
545 attribute_count: integer
546 Number of attributes that we track per particle (scalar values added
547 pre particle). Can be 0 if you don't want any attributes.
548
549 h and noise:
550 See tracer.InsertParticles
551
552 plot: Boolean
553 If this flag is set, ExaHyPE dumps the particles as vtu files whenever
554 it writes patch files. You can switch this behaviour off. A lot of codes
555 do so if they dump the tracer data independently into another database
556 anyway.
557
558 sort_descend_invocation_order_within_action_sets: Integer
559 You have to clarify what the priority of the tracer is within the
560 action sets. By default, all ExaHyPE solvers start with priority
561 0 and then up to n. So you are fine with using the default of -1
562 which means that the tracer precedes any solver. However, there
563 might be cases where you wanna move the priorities manually.
564
565 Returns the particle set that you can use to modify further
566 """
567 particle = peano4.toolbox.particles.Particle(name)
568 if isinstance(particle_attributes, dict):
569 self._particle_unknowns = sum(particle_attributes["unknowns"].values())
570 self._particle_variable_names += list(particle_attributes["unknowns"].keys())
571
572 for var in particle_attributes["unknowns"].values():
573 self._particle_variable_pos.append(var + self._particle_variable_pos[-1])
574
575 self._use_particle_var_shortcut = True
576 self._generate_particle_shortcuts(name)
577
578 elif isinstance(particle_attributes, int):
579 self._particle_unknowns = particle_attributes
580
581 else:
582 raise Exception(
583 "Invalid format for 'unknowns'. Must be an integer or a dictionary."
584 )
585
586 particle_attr = peano4.dastgen2.Peano4DoubleArray(
587 "data", str(self._particle_unknowns)
588 )
589 particle.data.add_attribute(particle_attr)
590
591 particle.data.add_attribute(
592 dastgen2.attributes.Enumeration("MoveState", ["NotMovedYet", "Moved"])
593 )
594
595 particle_number = peano4.dastgen2.Peano4IntegerArray("number", "2")
596 particle.data.add_attribute(particle_number)
597
598 particles = peano4.toolbox.particles.ParticleSet(particle)
599
600 self._project.datamodel.add_global_object(particle)
601 self._project.datamodel.add_vertex(particles)
602
603 self.plot_solution.use_vertex(particles)
604 self.checkpoint_solution.use_vertex(particles)
605 self.init_grid.use_vertex(particles)
606 self.perform_time_step.use_vertex(particles)
607 self.create_grid_with_refinement.use_vertex(particles)
608 self.create_grid_without_refinement.use_vertex(particles)
609
610 #
611 # Initialisation
612 #
613 self.init_grid.add_action_set(
614 peano4.toolbox.particles.api.UpdateParticleGridAssociation_LiftDrop(particles)
615 )
616
617 update_particle_association_action_set = (
618 peano4.toolbox.particles.api.UpdateParticleGridAssociation_LiftDrop(particles)
