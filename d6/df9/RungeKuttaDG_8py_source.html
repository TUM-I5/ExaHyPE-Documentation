39class RungeKuttaDG(object):
40 r"""!
41 An abstract class for any RKDG solver of any order
42
43 This is the base class of all Runge-Kutta DG solvers. It defines what kind of action
44 sets do exist, i.e., what in principle can be done while we run through the grid. It
45 also provides some kind of very basic infrastructure, i.e. what the name of a solver
46 is, what data is to be held per face or cell, or where in the multiscale mesh
47 we actually have to hold data.
48
49 The RungeKuttaDG class cannot/should not be instantiated. Its children actually
50 decide which steps are required or invoked in which order.
51
52 They do so by setting the guards: Each data structure has a guard which controls
53 if it is to be stored. Each action set has a guard that says if it should be
54 called in this particular mesh traversal.
55 If you want to redefine/reset these guards,
56 you have to redefine create_data_structures() and reset the guards after
57 you have called the superclass' operation. I recommend that you refrain from
58 defining completely new guards. Use the predefined guards instead and
59 refine them by adding more and and or clauses.
60
61 If you want to study the mathematics routines used by all the Python-generated
62 code, read the Discontinous Galerkin page in dg.h.
63
64
65 ## Data structures
66
67 Each cell hosts a DG polynomial of order _dg_order. Technically, we store this polynomial
68 in a regular array (blockstructured data) with (_dg_order+1)^d entries. On top of this
69 current time step, we need the shots into the future. for RK(_rk_order), we need _rk_order
70 of these snapshots. They are called estimates.
71
72 On the boundary, I store both the left and right solution projections. Per RK step, we
73 project the linear combination onto the face. This is in line with the action set
74 ComputeFinalLinearCombination which simply copies over the old time step
75 into the linear combination field in the very first Runge-Kutta step.
76
77 Our DG implementation is not tied to one particular Riemann solver, even though most
78 users will likely use Rusanov. Therefore, it is not clear which data is to be projected
79 onto the faces. For plain Rusanov with only a flux function, the solutions are sufficient.
80 Other Riemann solvers or Rusanov's ncp terms however require the derivatives. So I give
81 the code the opportunity to work with different data cardinalities.
82
83 More details are provided in create_data_structures().
84
85
86 ## Control flow between this class and subclasses
87
88 There are three key routines: the constructor, create_data_structures() and
89 create_action_sets(). The constructor sets/memorises some solver variables (such as the
90 name) and then invokes the other two routines.
91
92 create_data_structures() establishes all the data structures tied to the
93 grid entities. If you want to alter the configuration of data tied to grid
94 entities, you should redefine create_data_structures(). However, any
95 subclass still should call Runge Kutta's create_data_structures() before that.
96 This will ensure that the baseline configuration of all data is in place.
97 After that, you can modify the properties.
98
99 create_action_sets() establishes the action sets, i.e., activities that are to
100 be triggered whenever you run a time step, you plot, you initialise the grid.
101 Same here: If you want to alter the configuration of the code, call
102 create_action_sets() of the superclass first, before you do fancy stuff. This
103 way, all the important action sets are in place.
104
105
106 ## General control flow and realisation of solver steps
107
108 A proper numerical time step is mapped onto a cascade of time step sweeps. Per sweep,
109 we realise the following steps (so step in a computational sense), though their exact
110 orchestration depends on the realisation chosen, i.e. some might merge some
111 subcalculations into one grid sweep while others distribute a calculation over multiple
112 sweeps.
113
114 - We combine the data of the current time step with all the shots into the future. These
115 shots are held in one large array. This is not super efficient, as we could build up
116 these arrays one by one, but I just use one large scratchpad even for the first line
117 in the Butcher tableau. See create_data_structures() for details. So all the estimates
118 and the current solution are combined according to the Butcher tableau.
119
120 It is up to the particular subclass to configure the guard of the linear combination
121 such that it is only computed when it is actually needed.
122
123 - Next, we project this linear combination onto the faces. At the same time, we compute
124 a new estimate for the volume integral. The time step size for this new estimate
125 again depends on the time step size from the Butcher tableau. The two action sets
126 both are volumetric, i.e. run per cell, but we can run them in parallel. The face
127 projection has to complete immediately (we need the data on the faces), while the
128 volumetric computation, in principle, doesn't have to finish prior to the next grid
129 sweep, as long as the linear combination remains persistent. It is only a temporary
130 thing, so will be destroyed immediately once we leave a cell.
131
132
133 ## Mandatory step of subclasses
134
135 There are different nuances of subclasses/algorithmic realisations, i.e., few solvers inherit
136 directly from RungeKuttaDG. However, even those subclasses still require you do provide some
137 additional information.
138
139 These are the fields you have to set:
140
141 _compute_time_step_size A simple string which defines a double timeStepSize. This snippet will
142 be used by both the Riemann solver and the volumetric integration. You don't have to care
143 about the Runge-Kutta scaling of time step sizes (or shifts), as this is all done by the
144 source code, but you have to feed an instruction into the solver how to determine a time step
145 size for a cell.
146
147 self._compute_new_time_step_size = "const double newTimeStepSize = timeStepSize;"
148
149
150 ## Semantics of FaceLabel
151
152 - The update time stamp is the one we update in each and every
153 Runge-Kutta step.
154
155 - The new time stamp is the valid one, i.e., the one that
156 corresponds to the face's EstimateProjection after a time step.
157
158 - The old time stamp is the one that corresponds to the (old)
159 face projection.
160
161 - We set the updated flag if and only if we have finished the
162 last Runge-Kutta step.
163 """

443 coarsest_tree_level = 0

537 @abstractmethod
538 def create_data_structures(self):
539 r"""
540 Recall in subclasses if you wanna change the number of unknowns
541 or auxiliary variables. See class description's subsection on
542 data flow.
543
544 :: Call order and ownership
545
546 This operation can be called multiple times. However, only the very
547 last call matters. All previous calls are wiped out.
548
549 If you have a hierarchy of solvers, every create_data_structure()
550 should first(!) call its parent version. This way, you always ensure
551 that all data are in place before you continue to alter the more
552 specialised versions. So it is (logically) a top-down (general to
553 specialised) run through all create_data_structure() variants
554 within the inheritance tree.
555
556 :: Solver fields built up
557
558 _time_step: Patch ( (_dg_order+1)x(_dg_order+1)x(_dg_order+1) )
559 Actual polynomial data of the DG patch in the current time step.
560
561 _rhs_estimates: Patch ( (_dg_order+1)x(_dg_order+1)x(_dg_order+1)xno of RK steps )
562 These are the Runge-Kutta extrapolations. It is important that the
563 rk_order is mixed into the first argument, as the triple is
564 automatically truncated to a tuple if the code is translated with 2d.
565 Please note that the term rhs refers to the right-hand side of an ODE
566
567 @f$ \partial Q = F(Q) @f$
568
569 and thus does not include auxiliary variables.
570
571 _linear_combination_of_estimates: Patch ( (_dg_order+1)x(_dg_order+1)x(_dg_order+1) )
572 Non-persistent helper data structure.
573
574 _current_time_step_projection: Patch ( 2xface_projectionsx(_dg_order+1)x(_dg_order+1) )
575 This is the projection of the polynomial data of the current guess onto
576 the faces. It is important that the first dimension is only a 2. This
577 way, we can (logically) handle the projected data again as an overlap
578 of two patches with the width 1.
579
580 _estimate_projection: Patch ( Nxface_projectionsx(_dg_order+1)x(_dg_order+1) )
581 The N is usually 2. Usually this estimate projection holds the data
582 from the latest Runge-Kutta step. This implies that it never ever holds
583 the projection of the solution unless after the very first Runge-Kutta
584 step. After that, it is always a linear combination of estimates.
585
586 To avoid this plain behaviour, I make the last and final linear
587 combination project the solution again onto the faces, i.e. after
588 the very last step of the Runge-Kutta scheme, there will be a valid
589 representation of the solution _estimate_projection. Consult the
590 documentation of exahype2.solvers.rkdg.actionsets.ComputeFinalLinearCombination
591 for some further information. Also consult the semantics of the FaceLabel
592 in this context.
593
594 _old_solution_projection: Same as _estimate_projection
595 This is a backup of the final solution stored in _estimate_projection.
596 It correlates to
597
598 _face_label: FaceLabel
599 See class description. General information such as "is boundary".
600
601 _cell_label: CellLabel
602 See class description. General information such as "is enclave".
603
604 Per default, I do always store the projections onto faces, and I
605 always keep the actual time step data and the cell projections. It is
606 up to the subclasses to alter this storage behaviour.
607
608 By default, I do not exchange any face data in-between two grid sweeps.
609 You will have to overwrite the behaviour in subclasses. Please note that
610 a face is communicated between two tree/domain partitions if and only if
611 it is stored persistently and the exchange flag is set, too.
612 """

1317
1318 - Update of the labels: This can be done as very first step, as it
1319 might feed into follow-up steps.

1321 scheme. Without this first step, there's absolutely nothing we can
1322 do properly.
1323 - Couple resolutions and AMR: Once we have the linear combination,
1324 we can couple different resolutions.
1325 - Project linear combination onto faces: This should be done as soon
1326 as possible but after we've determined the linear combination. After

1329 the faces.
1330 - Handle boundary: doesn't really matter when we insert it, as it plugs
1331 into the first face load, while all the other stuff so far are volumetric

1333 It is important that we do the inter-grid transfer operators before we
1334 apply the boundary conditions.
1335 - Solve Riemann problem: the constraint here is that it has to come after
1336 the boundary handling.
1337 - Add volume and face solution: last
