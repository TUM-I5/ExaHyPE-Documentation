24 """!
25
26 Add action sets to step (aka mesh traversal) to adopt mesh to particle configuration
27
28
29 The AMR logic is based upon the observation that Peano's toolbox always
30 holds the particles in the right place and hence allows us to realise
31 all particle-particle interactions. If we don't get the mesh right, we
32 might not be optimal in terms of cost. But we are never wrong!
33
34 ## Straightforward workflow
35
36 In principle, SPH's AMR logic is pretty simple in the context of a the dual
37 tree with its lifts and drops. The illustration below shows this trivial
38 workflow on the left side:
39
40 @image html AMR-logic.png
41
42 The image is the time axis and the bullets are the individual mesh
43 traversals. On the left, we see the user input which eventually triggers a
44 cascade of actions on the bookkeeping side (right). The user alters the
45 particle position and hence might invalidate the particle-grid association.
46 We have to re-sort. Some sorting can happen right in the same mesh traversal
47 in which we alter the particles' properties in the touchCellLastTime or
48 touchVertexLastTime actions. The green colour denotes stuff that's happening
49 in the last data access - which might also introduce some MPI sends.
50
51 In the subsequent mesh traversal, we have to receive MPI data, move particles
52 down in their tree and associate that to the correct vertex. This happens
53 prior to any operation (denoted blue). Now that particles are in the right
54 place, we can update their parallel state.
55
56 A straightforward implementation now might evaluate the AMR criterion, as
57 all particles are in place and we can do so. Once we have collected all AMR
58 info, we can distribute it globally and hand it out in the subsequent mesh
59 traversal in which we consequently trigger refinements. Peano does not
60 realise refinements immediately, but waits for the next mesh traversal to
61 do so.
62
63 When we have done the refinement, we have to resort. Again, the resort has
64 to spread over two mesh traversals, as the first half of the sorting might
65 trigger MPI exchanges and multilevel traversals.
66
67 In this native data flow scheme, the user action affects the behaviour of
68 five follow-up mesh traversals.
69
70
71 Alternative to add_dynamic_mesh_refinement_and_particle_resorting_aggressive_mesh_adoption()
72 which is compatible with memory pool solutions, as long as you insert a dummy mesh sweep
73 after each routine which changes the particle-mesh topology (or you ensure that the
74 subsequent sweep does not evaluate particle-particle interactions).
75
76 The big difference to add_dynamic_mesh_refinement_and_particle_resorting_aggressive_mesh_adoption()
77 is that we do analyse the AMR after we've resorted the particles. However, we
78 do not realise the AMR up until we have to due to physics. So we make the AMR
79 lag behind even further.
80
81 The big differences hence arise in the section where we do the association, and
82 then we introduce the additional gather routines. As we do not update the
83 association four times but only twice, we can also update the state only once
84 instead of two times.
85
86 ## Workflow optimised towards minimal resorting
87
88 The optimised workflow accepts that we we are fine to deal with a suboptimal
89 mesh layout, as long as we minimise the number of particle movements. This
90 is a different cost objective than in the description above. We realise that
91 we will always need a resort when we have updated a mesh.
92
93 The idea is that we know that a user interaction forces us to update the
94 particle-mesh association, while AMR does not: We have to lift particles
95 from areas of the tree which are coarsened, but otherwise AMR does not
96 invalidate a valid mesh association. Particles might reside on too coarse
97 levels, but still all the kernel evaluations remain correct. The lift is the
98 only thing that causes issues, as it might mean that particles literally
99 travel.
100
101 To avoid that we have to do two sorts per particle position update, we
102 evaluate the AMR criterion two grid sweeps ahead of the actual particle
103 position update. This way, the AMR realisation and the actual position
104 update coincide in the same mesh sweep. No need for an additional sorting
105 step.
106
107 The disadvantage of this scheme is that the mesh always lags behind. The
108 AMR just is about the realise when the particles move again. So it might
109 have been optimal before, but now it is not good anymore. In return, we
110 however get a scheme which does, amortised, need only one resorting step
111 per mesh update plus AMR update.
112
113
114 ## Interplay with gathering
115
116 The aggressive AMR is not compatible with particles that gather data in memory
117 pools.
118
119 ## Realisation
120
121 This implementation puts emphasis that the mesh is always as tighly
122 adopted to the particles as possible. If necessary, this might mean
123 that we resort the particles multiple times per grid sweep: If they
124 change their position, we resort. But once we have resorted, we might
125 want to change the mesh, which in turn means we have to resort again,
126 as we might have new cells or cells might have been deleted.
127
128
129 ## Order of action sets
130
131 As so often, the order of the action sets is tremendously important,
132 as the individual action sets interact with each other. When we
133 discuss the order here, we may assume that all the function steps
134 aka action sets are already fully implemented and in the right order.
135
136 The trigger the AMR can be integrated whereever we want within the
137 pipeline. It simply deploys events, i.e. it is order-agnostic.
138
139 For the remaining action sets, we observe the following properties:
140
141 - Mesh analysis: Looks into the cell stats. These are in place, so
142 it should be one of the last things. The analysis happens in
143 touchCellLastTime().
144 - UpdateParallelState sets the particle flags in touchCellFirstTime().
145 In the last access to the vertex, it looks at all particles' flags
146 and throws away particles which are virtual. It is iportant that
147 that flag update happens when all particles are properly in place.
148 The throwing away should be the very last thing, though we can
149 throw away particles before we move them around if we want.
150 - UpdateAssociation. Should be the very first thing, but should
151 happen, before we update the state. In principle, the state can
152 also come later, but we might mess up the stats if it comes
153 before the particle migration: it might count particles multiple
154 times.
155
156 So we want to have the following lifecycle from a grid's point of
157 view:
158
159 - touchVertexFirstTime():
160 - Sort in the particles from coarser meshes or the global sieve list.
161 - Set their new parallel state.
162 - Then the user-defined stuff.
163 - touchCellFirstTime():
164 - Mark local particles as local, but it doesn't really matter when
165 this happens.
166 - touchCellLastTime():
167 - User code.
168 - Maintain some mesh statistics.
169 - touchVertexLastTime():
170 - User-defined stuff.
171 - Resort particles (to coarser levels, e.g.).
172 - Throw away virtual (remote) particles.
173
